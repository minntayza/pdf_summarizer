# agentic_workflows_M3_learner — Summary

- **What are Tools?**
  - Tools are functions/code that an LLM can request to be executed to extend its capabilities (e.g., getting current time, web search, calculations).
  - LLMs can dynamically choose tools when appropriate, based on user queries.

- **Simple Tool Execution**
  - Example: An LLM lacking time information can call a `get_current_time()` function via a tool, then respond with the result.
  - The LLM does not execute code itself; it returns a structured request (e.g., `FUNCTION: get_current_time()`), and the system calls the function and passes the output back.

- **Multiple Tools & Chaining**
  - LLMs can use multiple tools in sequence to accomplish complex tasks (e.g., check calendar for free slots, then make an appointment).
  - Tool calls can be chained: output of one tool (e.g., free times) used as input to another (e.g., create meeting).

- **Creating Tools – Prompting Approach**
  - A tool is just a Python function with a docstring. The system prompt tells the LLM about available tools and the exact syntax to invoke them (e.g., `FUNCTION: function_name(args)`).
  - Example: System prompt instructs LLM to return `FUNCTION: get_current_time("timezone")` when asked about time in a timezone.

- **Tool Syntax – Modern Frameworks (aisuite, OpenAI)**
  - Libraries like `aisuite` automate tool description: pass Python functions as `tools` parameter; the framework converts them to JSON Schema (name, description, parameters).
  - Behind the scenes, the function's name and docstring are used to create a JSON schema that tells the LLM about the tool.
  - Parameters (e.g., timezone) are automatically typed and described.

- **Code Execution as a Tool**
  - Instead of defining many arithmetic tools, the LLM can write and execute arbitrary Python code (e.g., `math.sqrt(2)`).
  - System prompts can instruct the LLM to output code inside special tags (e.g., `<execute_python>...</execute_python>`), and the system executes it and returns output.
  - This allows flexible computation without predefining every operation.

- **Reflection with External Feedback**
  - If code execution fails (syntax errors, runtime errors), the LLM can reflect on the error and produce an improved second draft.

- **Secure Code Execution**
  - Executing arbitrary code is risky; sandbox environments are recommended to prevent catastrophic errors.

- **Model Context Protocol (MCP)**
  - Problem: Every app builds its own tools, leading to m×n integration complexity.
  - Solution: MCP standardizes how tools are defined and discovered. Apps expose tools via shared MCP servers, reducing complexity to m+n.
  - Pre-built clients (Cursor, Claude Desktop) and servers (Slack, GitHub, PostgreSQL) are available, enabling quick integration.
