# agentic_workflows_M3_learner — Key Exam Points

🔴 **High Priority (Must-Know for Exams)**
- **Definition of tools**: Functions/code that LLMs can invoke to extend capabilities beyond their training (e.g., real-time data, calculations).
- **Two main patterns for tool integration**:
  1. **Prompting-based**: LLM outputs a special token (e.g., `FUNCTION: name(args)`) and the system executes the function externally.
  2. **Framework-based** (e.g., `aisuite`): Automatically convert Python functions to JSON schema and let the LLM decide when to use them.
- **Tool chaining**: LLMs can call multiple tools sequentially, using outputs of one tool as inputs to another (e.g., check calendar → make appointment).
- **Code execution tool**: LLM writes Python code inside tags, system executes it, returns result – flexible but requires sandboxing.
- **Reflection with errors**: If code fails, LLM can revise based on error messages (self-correction loop).
- **MCP (Model Context Protocol)**: Standard protocol to simplify tool integration across apps; reduces integration complexity.

🟡 **Medium Priority (Important to Understand)**
- **Tool creation via function docstrings**: The docstring becomes the description sent to the LLM – crucial for correct invocation.
- **JSON schema generation**: Parameters, types, and descriptions are automatically derived from function signatures and docstrings.
- **Examples of tools**: `web_search`, `interest_calc`, `query_database`, `get_current_time`.
- **Secure execution**: Always use sandboxes for arbitrary code execution to avoid security risks.
- **Difference between simple function tools and code execution**: Simple tools are predefined; code execution is dynamic and arbitrary.

⚠️ **Common Pitfalls & Mistakes**
- **Forgetting to include parameters correctly**: Tool calls must match the function signature (e.g., `get_current_time("Pacific/Auckland")` not `get_current_time`).
- **Not handling tool output properly**: The system must feed the tool result back into the conversation context for the LLM to continue.
- **Assuming LLM can execute code directly**: The LLM only *returns* a function call; the system must execute it.
- **Over‑reliance on code execution without sandboxing**: Arbitrary code can delete files or access sensitive data – always sandbox.
- **Misunderstanding MCP**: It standardizes tool definitions, not tool execution – still need a client and server to handle calls.
