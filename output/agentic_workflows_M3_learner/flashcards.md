# agentic_workflows_M3_learner — Flashcards

## Flashcard 1
**Q:** What is a 'tool' in the context of Agentic AI?
**A:** A tool is a function or piece of code that an LLM can request to have executed externally, enabling it to perform tasks it cannot do alone (e.g., fetching live data, computing math).

## Flashcard 2
**Q:** Describe the basic flow of a simple tool execution (e.g., get_current_time).
**A:** 1. User asks a question requiring real-time info.
2. LLM outputs a structured request like `FUNCTION: get_current_time()`. 
3. The system detects the request, executes the function, and gets the result (e.g., '15:20:45').
4. The system feeds the result back into the LLM context.
5. LLM uses the result to answer the user.

## Flashcard 3
**Q:** How does the LLM know which tool to use and how to invoke it?
**A:** The system prompt describes available tools and their exact invocation syntax (e.g., 'To use get_current_time, return `FUNCTION: get_current_time("timezone")`'). Modern frameworks automatically generate tool descriptions via JSON Schema from the function's name, docstring, and parameters.

## Flashcard 4
**Q:** What is the advantage of letting the LLM write and execute arbitrary Python code instead of having many specific tools?
**A:** It removes the need to predefine tools for every operation (e.g., arithmetic, sqrt). The LLM can write dynamic code to solve a wide range of computational problems, making the system more flexible and scalable.

## Flashcard 5
**Q:** Explain the 'reflection with external feedback' pattern for code execution.
**A:** After the LLM writes code, the system executes it. If the code produces an error (syntax, runtime), the error message is fed back into the LLM. The LLM reflects on the error and produces an improved version of the code, repeating until it runs successfully.

## Flashcard 6
**Q:** Why is secure execution important when using code generation tools?
**A:** Running arbitrary code outside a sandbox can cause catastrophic errors (e.g., file deletion, data leaks). Sandboxes isolate execution to protect the system and data.

## Flashcard 7
**Q:** What problem does the Model Context Protocol (MCP) solve?
**A:** Without MCP, each app must build custom tools for every external service, leading to m×n integrations (m apps × n services). MCP standardizes tool definitions and discovery via shared servers, reducing complexity to m+n.

## Flashcard 8
**Q:** Give an example of chaining multiple tools in a single user request.
**A:** User: 'Find a free slot on Thursday and make an appointment with Alice.'
1. LLM calls `check_calendar(day=Thursday)` → gets free times.
2. LLM calls `make_appointment(person=Alice, time=3pm)` → confirms success.
3. LLM then responds to user: 'Appointment set for 3pm Thursday.'

## Flashcard 9
**Q:** What information is automatically extracted from a Python function to create a tool description for the LLM?
**A:** The function's name, its docstring (becomes the description), and its parameters (names, types, and docstrings) are converted into a JSON Schema object that tells the LLM how and when to use the tool.

## Flashcard 10
**Q:** Contrast the old approach of many small tools (e.g., add, subtract) with the code execution approach for a calculator.
**A:** Old approach: need separate tools for each operation (add, subtract, sqrt, etc.). Code execution approach: only one tool that lets the LLM write arbitrary Python code to compute any math expression, reducing the number of tools and increasing flexibility.

## Flashcard 11
**Q:** What is a common mistake when designing system prompts for tool use?
**A:** A common mistake is not clearly specifying the exact syntax for function calls (e.g., the LLM might output plain text instead of `FUNCTION: name(args)`). Also, forgetting to describe parameter formats (e.g., timezone strings) can lead to incorrect invocations.

## Flashcard 12
**Q:** How does the `aisuite` library simplify tool usage compared to manual prompting?
**A:** `aisuite` automatically converts Python functions (with docstrings) into tool descriptions for the LLM, handles function call detection, execution, and result injection into conversation history without needing custom prompt parsing.
